# Cupid Toolchain bootstrap

This directory is the durable record for moving Cupid OS from its current host-produced build to a self-hosting Cupid Toolchain. The implementation map is [GitHub issue #13](https://github.com/cupidthecat/cupid-os/issues/13).

The shared frontend carries compatible structure and union values through plain assignment, return, automatic expression initialization, fixed arguments, and matching-record conditional expressions. It retains automatic, block-static, and file-scope array or structure initializer lists with direct C11 member and array designators. Selectors stay in source order, positional initialization resumes after the selected subobject, and a direct unknown-bound array uses its greatest selected index plus one. Brace-elided children return a following designator to the nearest explicit list. The audit finds 600 direct active-source designators across 14 files, split between 409 member selectors and 191 array selectors. The contract includes the sparse 134-byte ELF table from the CupidASM kernel object test and all seven 35-member definitions in unchanged `kernel/gui/gui_themes.c`. Chained designators, names promoted through anonymous members, duplicate overrides, and positional union or Cupid class lists remain deferred.

The production in-kernel CupidC emitter now sends `continue` in a `do` loop to the condition. It enters the body past a two-jump trampoline. The `continue` target lands on the trampoline, which is patched to the condition after the body has been emitted. `/bin/feature25.cc` runs this path through the GUI terminal and in-OS JIT; the checked marker is `[feature25] PASS body=2 condition=2`. A `continue` inside a `switch` nested in a loop is still a known private-emitter limitation because switches and loops share one depth stack.

The shared frontend independently publishes typed C11 `do` statements with a converted scalar condition and a required body. It records the body and its expressions before the condition and loop node. `break` and `continue` remain targetless until IR lowering, where a post-test `continue` must branch to the condition.

Block bindings and file object definitions own one job-owned semantic initializer forest. Automatic scalars and whole records use `EXPRESSION` roots. Automatic aggregate lists use `LIST` roots with runtime `EXPRESSION` leaves, while character arrays use `STRING`. Supported static objects use `ZERO`, target-converted `INTEGER`, `STRING`, string `ADDRESS`, an `ADDRESS` of a linked file object or function, or recursive `LIST` records. An explicit static null pointer constant uses a destination-typed `ZERO` record, including when it is a child of an array or structure list. A binding address names a linked file object or function and may carry a checked signed i386 target-byte addend. Pointer addition, subtraction, and subscripts derive that addend from the existing integer constant-expression value and the target referent layout. List edges name explicitly initialized direct array elements or structure members, nested roots are postorder, and omitted subobjects remain implicit zero. A direct unknown-bound array is completed on a private object type, leaving a shared incomplete typedef unchanged. Freeze derives storage duration from the owning definition or binding, accepts explicit `ZERO` children only in static forests, and rejects runtime leaves there. The forest does not serialize a target image, allocate static data, or emit relocations.

File-scope object definitions live in a table separate from canonical bindings. The binding keeps first-declaration facts, while the definition keeps its type, storage, source location, explicit or tentative kind, and initializer root. Repeated tentative declarations coalesce as they are parsed. Translation-unit finalization applies the merged binding type and supplies a zero root. An incomplete external array becomes a one-element array, while incomplete internal arrays and records fail precisely. The focused contract publishes 29 definitions, 39 initializer records, and nine list edges. It covers repeated and superseded tentative declarations, object addresses, array and function decay, array-element and one-past addresses, pointer subtraction, integer-first addition, `&numbers[1] + -1`, nonzero qualification and object-to-`void` conversions, both sides of the signed i386 addend boundary, unevaluated pointer arithmetic inside `sizeof`, unresolved external references, and mixed address leaves without inventing host pointers or ELF relocation records. Unchanged `kernel/fs/ramfs.c` proves the string and all 11 function addresses in its operations table.

The hosted object path now accepts represented static data and an initial leaf-function subset in the same deterministic i386 ELF32 relocatable object. Public `ctool_c_lower_ir` turns each supported function into a contiguous typed address/value instruction slice with function-relative branches and retained source locations. The first ABI slice covers fixed, nonvariadic cdecl functions with `void` or 32-bit integer results and 32-bit integer parameters. It lowers parameter loads, same-width implicit integer conversions, constants, subtraction, signed or unsigned greater-than comparison, conditional selection, and value or void return. Static inline definitions and non-inline definitions after inline declarations are accepted; external-inline definitions stop at the pending finalization boundary. `ctool_c_emit_object` places `.text` before `.rodata`, `.data`, and `.bss`, gives defined functions `STT_FUNC` symbols, and emits every instruction through the shared x86 encoder. Static initializer lists, direct-symbol `R_386_32` relocations, and exact addends 4 and 8 keep their earlier behavior. The mixed object contract decodes the emitted functions through the shared decoder, pins every machine byte including parameter displacements and operand order, resolves both conditional branch targets, checks `SETG` and `SETA`, and proves byte-identical repeat output. Calls, local objects, explicit casts, general statements and expressions, floating or aggregate ABI work, `R_386_PC32`, and production integration remain open. Invalid units, unsupported bodies, external-inline definitions, and output limits still leave output empty and rewind allocations made during the operation.

Active Doom declarations require the same array-address form at `kernel/doom/src/g_game.c` for `mousearray` and `joyarray`, and at `kernel/doom/src/tables.c` for `finecosine`. The focused contract mirrors the constant-expression subscript used by `finecosine`. A full exact-profile parse still stops earlier at `kernel/doom/dglibc_compat.h` because `__builtin_va_list` is not yet represented, so this evidence does not claim Doom translation-unit ownership.

- `../adr/` records stable architectural decisions; `../../CONTEXT.md` defines project vocabulary.

## Update contract

Every toolchain implementation commit must update the affected records here and include relevant positive and negative tests. Claims in these files must distinguish source inspection from executed verification. The `TempleOS/` reference tree is excluded from all progress metrics. Generated objects, images, and logs are excluded from source-migration counts and ordinary commits unless they are intentional bootstrap inputs such as checked seeds; their hashes, ownership, layout, and runtime behavior remain required acceptance evidence.

Progress means transferring ownership without reducing Cupid OS behavior:

1. A Cupid tool gains the real feature required by an active source cohort.
2. Tests prove successful behavior and useful failures.
3. The cohort moves from the legacy host/oracle path to the Cupid path.
4. The OS build and applicable boot smoke tests remain green.
5. Host dependencies are removed from the normal build only after the replacement path is proven.
