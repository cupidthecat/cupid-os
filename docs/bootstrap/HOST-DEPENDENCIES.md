# Host dependency inventory

The deterministic active-source audit records three supported build roots: root `all`, `user:all`, and `toolchain:all`. `audits/active-build.json` owns the current 681-source/490-transform graph and its 431-path manifest covers all 424 final-link objects. The language graph contains 267 C translation units, 262 headers, and 126 Cupid C files; the prioritized toolchain-source cohort contains 53 files. The checked Windows Clang/LLVM and Linux GCC/binutils baselines both reproduce the complete 447-artifact three-root cohort at revision `1e079d1`; `baselines/windows-linux.json` verifies their common logical cohort and required behavior while treating cross-toolchain byte equality as observational. That evidence predates the CupidC preprocessing, declaration, and type/layout contracts. The current graph declares sixteen hosted artifacts, so its complete three-root cohort would contain 450 logical artifacts when recaptured from one committed revision.

`tools/hostbuild.py::_symbols_from_nm` remains the symbol-reader subprocess seam. `embed_jpeg` performs optional image preprocessing, then calls CupidObj once with the original source identity; the former temporary-name wrapper plus three-symbol rewrite pass was removed. `tools/mksyms.sh` and `tools/embed_jpeg_baseline.sh` are tracked legacy/oracle duplicates outside the normal Make path.

The hosted contract suites use the host C compiler and its native linker backend to bootstrap and exercise the shared core, CupidC preprocessing/declaration/type-layout operations, ELF32, x86, CupidDis, CupidASM, CupidObj, CupidLD, and the kernel's buffer-only fixed-image-to-`ET_EXEC` bridge. The ELF32, CupidASM, and CupidLD suites may additionally use NASM, GNU `readelf`, and standalone GNU/LLVM ELF linkers as optional comparison oracles. They prove that Cupid-written objects and executables are accepted by external consumers, that the Cupid reader accepts Clang-, NASM-, and linker-produced objects, that every active assembly source reaches the required raw, relocatable, or fixed artifact, and that all shared operations fail transactionally; absent oracle tools are skipped. Assembly, inspection, object-transformation, and link semantics plus production assembly/link/object ownership have transferred.

The shared preprocessor owns translation phases, macros, conditionals, predefined macros, dual-location `#line`, includes, once identity, pack metadata, and policy-neutral Cupid `#exe` markers. The declaration operation consumes that tape as its sole lexical input and publishes the shared type graph, completed layouts, canonical bindings, tags, normalized parameters, first-declaration storage/provenance, effective linkage, packed/aligned layout effects, function `noreturn`, minimum entity alignment, names, and dual locations. The unchanged FAT16 closure reproduces every manual layout oracle, while four unchanged redeclaration oracles, unchanged attribute ABI/effect oracles, and the 146/152 active non-Doom header sweep prove compatible declarations beyond it. Alignment-transparent composites retain incomplete/fixed bounds plus recursive `const`/`_Atomic` identity without depending on host layout. Dynamic checked storage removes the private parser's fixed typedef/member caps; the public 256-level limit governs recursively nested source syntax, while iterative memoized type walks accept a contracted 512-level derived pointer graph. The declaration subset still has linear namespace/member lookup, a partial integer-expression grammar, no tentative-definition state, and no static assertions, bodies, initializers, or remaining GNU attributes. The host compiler builds all three modules, the kernel frontend consumes none of them, and full expression/statement checking, AST/IR, code generation, execution policy, and source-compilation ownership remain host-bound. These capabilities retire no host dependency by themselves.

The hosted preprocessor contract loads and preprocesses all 345 tracked OS-delivery translation-unit roots through the repository file adapter under GNU-Make-evaluated profiles. This covers 238 tracked root-kernel/Doom C inputs, three user inputs, and 104 Cupid programs; a separate target materializes and requires the four generated kernel C inputs through the existing generators and first-pass link. This is compatibility evidence, not host-compiler retirement: the host compiler still compiles 245 root/user objects and the hosted contracts. Of the 28 hosted C translation units deferred from the OS-delivery gate, 17 still require external system headers/runtime and eleven hermetic units need a defined hosted Cupid profile before they can join without importing platform header trees.

The hosted layout contract still supplies independent manual ABI oracles for every FAT16 member offset, active Doom bit fields, and representative process, syscall-table, `e1000_rx_desc_t`, and per-CPU layouts. The declaration contract now supplies the separate source-driven proof for the FAT closure plus namespace, declarator, declaration-legality, target-integer, rollback, scale, and nesting cases. The declaration contract is the sixteenth host-built artifact. Neither it nor the layout artifact affects an OS binary or justifies an emulator result; the next ownership gate is additional unchanged active declarations followed by typed-AST/IR and object lowering.

The tracked `link.ld` is itself a compatibility contract. It uses `ENTRY`, `SECTIONS`, location-counter assignment, input-section wildcards, `ALIGN`, symbol definitions, `COMMON`, and repeated `ASSERT` statements. Both kernel ELF targets declare it as a prerequisite and pass it to CupidLD.

## Not host compilation

- The 104 active `bin/*.cc` roots and 22 `bin/browser/*.cc` fragments are wrapped by CupidObj and installed in the OS filesystem. CupidC compiles them on demand inside Cupid OS.
- The 22 `demos/*.asm` files are likewise embedded by CupidObj and assembled by CupidASM on demand.
- Repository headers and compatibility code replace the host libc/header environment for root compilation (`-nostdlib -nostdinc -ffreestanding`). The compiler executables remain external.
- The hosted contracts intentionally use the host C runtime only through the core adapter and thin CLI drivers. The shared arena, buffer, path, source, diagnostic, limit, object, instruction, assembly, and inspection behavior is freestanding and the same CupidASM source is linked into the kernel.
- Optional WAD discovery and test fixtures affect packaged/runtime content, not compiler ownership.

## Removal gate

A code-producing host dependency leaves the normal build only after the Cupid replacement has positive and negative tests, matches required object/ABI/layout behavior, builds its assigned active-source cohort, and passes the relevant OS boot or runtime smoke. The legacy host path remains available as an oracle until fixed-point bootstrap and behavior gates are reliable.
