# Cupid Toolchain capability matrix

The matrix uses these statuses:

- **Observed**: the implementation is present in source and at least one relevant smoke was executed.
- **Partial**: useful implementation exists, but the required active-source or format coverage is not proven.
- **Missing**: no implementation satisfying the bootstrap contract was found.
- **Required**: the checked active-source audit proves the capability is exercised, but the Cupid implementation does not yet satisfy it.

The capability requirements below are backed by `ACTIVE-SOURCE-AUDIT.md` and `audits/active-build.json`: 688 active language inputs across root, user, and hosted toolchain build roots, 251 stable feature IDs, 498 output transforms, and an explicit i386 ILP32/cdecl/ELF32 contract. The language split is 26 assembly files, 271 C translation units, 264 C headers, and 127 Cupid C files. Lexical counts are discovery evidence; semantic completion still requires focused compiler and assembler tests.

Explicit casts to `void` now evaluate represented integer, object pointer, function pointer, supported structure, and `void` operands without producing a value. A represented scalar or structure emits typed `DISCARD`; a `void` expression leaves the abstract stack at its incoming depth. A structure is first captured in an instruction-owned snapshot, so discarding it does not weaken the normal lvalue conversion. The unchanged `ctool_host_allocate` and `ctool_host_release` helpers guard the original active requirement. The aligned scalar object proof has one 52-byte function, three symbols, and one direct-call relocation at text offset 43. Wide scalar, floating, union, Cupid class, array-value, volatile, and atomic operands still stop at their existing evaluation boundaries.

Automatic initializer lists for complete fixed arrays and structures reach hosted IR and deterministic object emission. One `ZERO_OBJECT` applies the aggregate's implicit-zero rule, then source-ordered explicit leaves store through direct `MEMBER_ADDRESS` and `ELEMENT_ADDRESS` paths. A represented structure-valued leaf uses the structure copy path. A narrow character-array string leaf uses `COPY_STRING` after the complete root has been zeroed, so bytes beyond the retained source stay zero. The i386 emitter preserves EDI, establishes a forward direction with `CLD`, and uses `REP STOSB` for zeroing plus `REP MOVSB` for string copies. Active guards pin the unchanged `no_name` and type-node initializers plus the automatic hexadecimal array in `drivers/serial.c`. Explicit bit fields, union or Cupid class values, volatile or atomic subobjects, and wide or floating scalar expression leaves remain unsupported in this path.

Block-scope compound literals reach the same hosted IR and object path. One absolute expression identity names the persistent unnamed automatic object at each source site. Its initializer runs at every evaluation, and `COMPOUND_LITERAL_ADDRESS` leaves the object's address as the lvalue result. Aggregate lists build in a separate slot through `COMPOUND_LITERAL_STAGING_ADDRESS`, then `COPY_OBJECT` commits the complete value after every initializer read has finished. A narrow `STRING` root zeroes its persistent array and copies the retained bytes directly. The audit records 40 compound-literal occurrences across four active files. Contracts emit the unchanged `(ctool_string_t){literal, size}` call and `(char[]){"Cupid"}` deterministically. Static-duration compound literals, variable-length literal objects, and the named-aggregate backward-jump alias case remain open.

Complete structure values reach hosted IR and deterministic i386 object emission. The abstract stack keeps one handle for each structure value. `LOAD` captures an lvalue in an instruction-owned frame snapshot, `STORE` copies without a result, and `STORE_VALUE` preserves the source snapshot for assignment chains. Structure values can cross compatible conditionals, whole-record expression initialization, casts to `void`, fixed direct and indirect calls, and returns. The cdecl emitter copies arguments inline in rounded four-byte spans, zeros ABI padding, and uses a hidden result pointer for a structure return. A structure-returning callee reads that pointer at `EBP + 8`, starts explicit parameters at `EBP + 12`, returns the same pointer in EAX, and removes the hidden slot with encoded `RET 4`. The caller removes the explicit outgoing spans and its temporary argument handles after the call. Structure snapshots and call results receive deterministic frame temporaries after ordinary locals. This path accepts complete `struct` objects whose inline graph is nonvolatile, non-atomic, and aligned to at most four bytes. A bytewise structure may contain fixed arrays, bit fields, wide integers, or floating members. Unions, Cupid classes, array values, over-aligned objects, structure-valued ellipsis arguments, and floating calls remain open. Every supported call now aligns ESP to sixteen bytes before `CALL`, including direct and indirect structure calls. No production C object uses this hosted path.

Scalar variadic calls and callees now cross the shared frontend, Linear IR, and i386 emitter. The caller applies represented default argument conversions, retains its actual argument count, and uses that count for cdecl order, indirect callee placement, sixteen-byte alignment, and cleanup. In GNU C mode, `__builtin_va_list` is a target `char *` cursor. Explicit start, argument, copy, and end operations let a definition traverse non-atomic four-byte `int`, `long`, and pointer slots after its final named parameter. The active audit records 49 variadic declarations. Atomic cursors and reads, floating default promotions, non-scalar ellipsis transport, and wider or aggregate reads remain open.

Empty identifier-list definitions and unprototyped calls now cross the same hosted path. A definition written with `()` has zero parameters and keeps its non-prototype function type. Every argument at an unprototyped call site receives the default argument promotions, while Linear IR retains the actual count for direct or indirect cdecl emission. The represented ABI accepts four-byte integers and pointers at this boundary. Nonempty identifier lists, floating promotion to `double`, non-scalar arguments without declared parameter types, and production ownership remain open.

`ctool_c_parse` publishes compatible structure and union values through plain assignment, function return, automatic expression initialization, fixed arguments, and matching-record conditional expressions. It represents C11 null pointer constants with destination-typed `CTOOL_C_CONVERSION_NULL_POINTER` nodes in runtime AST contexts. Block bindings and file object definitions own a job-owned semantic initializer forest. Automatic scalars and whole records use `EXPRESSION` roots. Static objects use `ZERO`, target-converted `INTEGER`, copied narrow-array `STRING`, string `ADDRESS`, an `ADDRESS` of a linked file object or function, or recursive `LIST` records. Explicit static null pointer constants use destination-typed `ZERO` records, including aggregate leaves. File definitions retain definition facts separately from the canonical binding, and tentative definitions receive zero roots during translation-unit finalization. A binding address names a linked file object or function and may carry a checked signed target-byte addend from pointer arithmetic or a subscript with a represented integer constant expression. List edges retain direct member or array selectors in source order, including C11 designators and following positional clauses. Omitted subobjects remain implicit zero without an edge. Unknown-bound arrays complete from the greatest selected element plus one without changing a shared typedef. The forest records source semantics only. The separate `ctool_c_emit_object` operation assigns static storage, emits bytes and symbols, and lowers represented addresses and addends to `R_386_32` relocations. A bounded iterative query still enforces the C11 modifiable-record rule across complete records, nested records, unions, and arrays.

The object operation now consumes static roots owned by file definitions and block bindings. Block-static declarations emit no runtime initializer instructions. Their `LOCAL_ADDRESS` identities map to local ELF symbols and `R_386_32` relocations instead of automatic frame slots. File objects retain first placement, block statics follow in absolute binding order, and functions come afterward.

The generated active-source audit records the current lexical totals and build graph. All nine hosted Toolchain source gates parse completely. Each five-number tuple reports definitions, statements, expressions, block bindings, and initializers. `cupidc_pp.c` publishes 143/3,904/25,107/475/282. `cupidc_ir.c` publishes 161/4,936/43,159/607/196. `cupidc_emit.c` publishes 122/3,102/27,793/416/209. `cupidc_frontend.c` publishes 299/11,675/75,085/1,731/1,189. The hosted IR and object slice changes no production owner or runtime ABI. The private-emitter fix changes one production JIT branch target and is covered by an in-OS smoke.

Hosted function-pointer compatibility uses checked job scratch and a memoized worklist. It ignores top-level parameter `const`, `volatile`, and `restrict`, retains `_Atomic` and referent qualification, and applies the existing old-style default-promotion rule. A 24-level repeated callback graph pins the nonrecursive relation and constrained-arena rollback. Explicit function pointer casts that produce values are still unsupported. Casting a represented function pointer to `void` discards it without defining a target conversion. Represented integer casts cover one-byte, two-byte, and four-byte integers, while the existing object-pointer cast path remains four-byte. A separate 28-byte object proves an absolute relocation to a defined static function.
