# Toolchain ownership migration matrix

`TempleOS/` is excluded: it is reference material, not a source cohort. Statuses describe ownership, not how much code exists.

Hosted CupidC now parses all twelve hermetic `HOSTED_TOOLCHAIN_64` implementation files and emits deterministic i386 ELF32 objects for those files plus `kernel/lang/as_elf.c`. This supersedes the older nine-file counts in the historical notes and long-form rows below. Eleven of issue #27's fourteen CupidC, CupidASM, and CupidDis files reach the object gate. `ctool_host.c`, `cupidasm_main.c`, and `cupiddis_main.c` remain outside it because they need external host headers, runtime support, and a host-runnable ABI. The new source forms are nested union storage inside supported structures, scalar members from structure rvalues, a direct four-byte literal-zero function-pointer cast, object-pointer conversion to and from signed or unsigned eight-byte integers, and parenthesized static string addresses. This work moves no production transform or host dependency. ADR 0081 records the exact boundary.

Eight-byte integer values expand hosted CupidC capability without moving a production transform. Constants, matching conditional arms, fixed direct and indirect call results, object loads, initialization, plain and chained assignment, all ten compound assignments, prefix and postfix update, declared parameters, named call arguments, discarded values, returns, addition, subtraction, multiplication, division, remainder, unary plus, unary minus, bitwise complement, shifts, AND, OR, XOR, all six comparisons, logical not, short-circuit logical operators, conditional selection, structured scalar conditions, signed and unsigned switch dispatch, explicit represented-to-wide casts, same-rank signed-to-unsigned conversion, GNU wide-enum promotion, and conversion to or from represented integer widths use one Linear IR handle backed by an emitter-owned eight-byte snapshot. File objects, block statics, fixed automatic objects, pointer dereferences, ordinary members, and indexed elements reach that path. A declared argument receives eight cdecl stack bytes, and the return boundary places the low word in EAX and the high word in EDX. Calls retain packed post-conversion actual types in emitted call order. Direct and indirect ellipsis or unprototyped calls pass each signed or unsigned wide integer in adjacent low and high i386 words. A wide variadic read advances its cursor by eight bytes and returns one snapshot handle. Wide switch dispatch duplicates the snapshot handle and compares complete case values without evaluating the condition again. Mutation evaluates its destination once and uses the existing wide operation and snapshot-store paths. Multiplication combines one full low-word product with both cross-word products in a fresh snapshot. Division and remainder use a fixed 64-step restoring loop over private stack scratch; signed operations divide magnitudes, then apply the quotient sign from both operands or the remainder sign from the dividend. Focused proofs guard the unchanged `cfront_integer_mask`, `get_cpu_freq`, `ctool_buffer_put_le64`, `ctool_buffer_patch_le64`, `pp_if_value_truth`, `pp_if_is_negative`, `pp_if_signed_less`, `pp_if_signed_magnitude`, CupidASM number-parser and unary-expression requirements, the frontend's quotient/remainder logic, and X25519's `fe_mul_u32` and `fe_carry` helpers. The `decode_function` and `validate_file_object_finalization_storage_limit` guards cover real signed and unsigned wide `fprintf` arguments. They guard source fragments, not whole-function ownership. No active-source guard covers a wide variadic read or an unprototyped wide call, so those paths remain focused ABI fixture evidence. GCC or Clang still builds the compiler and every normal C object, so ownership and dependency counts do not change.

The quotient/remainder object contains eleven functions and thirteen software loops in 4,775 text bytes. This work does not move production ownership. The shared frontend, Linear IR, and deterministic i386 emitter move matching `float` and `double` values through objects, assignment, calls, variadic reads, and returns. Hosted structure values also leave ownership unchanged.
