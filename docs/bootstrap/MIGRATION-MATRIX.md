# Toolchain ownership migration matrix

`TempleOS/` is excluded: it is reference material, not a source cohort. Statuses describe ownership, not how much code exists.

The private production compiler sends `continue` in a `do` loop to the condition through a patched trampoline. `/bin/feature25.cc` proves the behavior in the in-OS JIT. This fixes code generation without changing ownership. The shared hosted frontend retains automatic, block-static, and file-scope initializer forests with direct member and array designators. It publishes explicit and tentative file object definitions and semantic addresses of linked objects or functions, including checked target-byte addends from constant pointer arithmetic and subscripts. The audit counts 600 direct designators across 14 active files, and unchanged `kernel/gui/gui_themes.c` proves 245 of them across seven file-scope definitions. Unchanged `kernel/fs/ramfs.c` proves all 11 function addresses in its operations table. The hosted object operation emits the represented static definitions and addresses through the shared ELF32 writer. All seven unchanged Toolchain source gates parse completely, including `cupidc_emit.c` and `cupidc_frontend.c`. A `continue` inside a `switch` nested in a loop remains a private-emitter limitation, and issue #25 stays open.
