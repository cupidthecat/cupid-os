# Use one typed x86 instruction seam and one private form catalogue

CupidC, CupidASM, and CupidDis will share a freestanding 16/32-bit x86 module whose public values are typed instructions, operands, encoded bytes, decoded results, and relocation-capable field spans. The semantic model carries execution mode, canonical mnemonic identity, lock/repeat intent, GPR8/GPR16/GPR32, segment, control, x87, MMX, and XMM registers, immediate and relative values, far pointers, and memory operands whose value width is distinct from address width, segment, base, index, scale, displacement, and requested serialized width. Encoding is deterministic and transactional into a caller-owned 15-byte value; every failure fully zeros that value. `AUTO` selects the shortest successful encoding and uses stable catalogue order to break equal-length ties. Decoding distinguishes known, unknown, truncated, and invalid input; it retains the exact consumed bytes and an opaque catalogue-version-scoped form identity so inspection can preserve a noncanonical spelling without pretending that semantic re-encoding will reproduce redundant prefix order or an equivalent opcode alias. A form identity is meaningful only with the model fingerprint that produced it, and a requested form must match the supplied instruction exactly. The fingerprint covers ordered form rows and the complete mnemonic and register name tables, including aliases and canonical markers, rather than only serialized opcodes.

One private flat catalogue is the authority for both directions. Each form declares legal modes and ISA level, mnemonic, operand recipes, mandatory and semantic prefixes, opcode map and bytes, opcode-plus-register behavior, ModRM register/rm roles or fixed digit, immediate/relative rules, and privilege, lockability, and control-flow metadata. Decode-only invalid rows use the same schema and authority for the small set of reserved or mode-illegal spellings the model recognizes; a mismatch against an ordinary row never by itself proves that otherwise unimplemented bytes are invalid. In this first seam those properties drive matching and validation only; they are not a public control-flow-analysis API, which will be added only when a real CupidDis or linker consumer establishes the required vocabulary. Private lookup indexes may be derived from those rows, but no second opcode table or family-specific encode/decode callback may become another authority. The initial catalogue is driven by the exact boot, kernel, embedded CupidASM, CupidC-emitted, and active inline-assembly surface; its public register and operand vocabulary is already able to grow through x87, MMX, and SSE2 without an interface change. x86-64, REX, VEX, EVEX, and later domains remain excluded by ADR 0005.

Encoded field metadata identifies operand, byte offset, width, absolute or PC-relative relocation intent, and PC bias while treating symbol references as opaque caller-owned values. The x86 module does not own parsing, expression evaluation, labels, symbol tables, ELF policy, text formatting, object inspection, executable layout, or JIT policy. A later CupidASM maps supported four-byte absolute and relative fields to the shared ELF32 `R_386_32` and `R_386_PC32` model; a later CupidDis overlays object relocations on decoded fields.

Catalogue name lookup is ASCII case-insensitive resolution of one mnemonic or register atom, not source parsing; aliases return a canonical mnemonic and returned names have static lifetime. `UNKNOWN` decoding means the bytes are outside the implemented catalogue and consumes one byte for inspector progress. `TRUNCATED` means a known prefix/opcode path needs more input and consumes zero bytes while retaining the available prefix bytes. `INVALID` means a recognized family has an illegal prefix, mode, or explicitly reserved encoding. Repeating a prefix from the same legacy prefix group is deliberately invalid in this seam, even when some processors tolerate it, because v1 preserves one semantic prefix choice rather than arbitrary prefix histories; a prefix run that reaches the architectural 15-byte instruction limit without an opcode is invalid as well. Prefix groups are decoded independent of their byte order, while semantic re-encoding uses one canonical order; for example, GNU `66 F3 6D` decodes as `REP INSW` and re-encodes as `F3 66 6D`. Raw decoding always reports constant serialized fields with no relocation ownership; only an object consumer can overlay relocation records.

A semantic-definition table plus interpreted bidirectional codec recipes was rejected for the first implementation. It compresses regular x86 families, but adds a recipe VM, collision/index construction, trace policy, and exceptional handlers before the measured corpus requires that complexity. Separate family callbacks were also rejected because opcode facts could drift back into procedural encoder and decoder paths. The flat catalogue intentionally accepts some repetitive rows in exchange for direct auditability, bounded matching, and a mechanically testable one-fact/one-row rule; parameter generation or recipe compaction can be reconsidered behind the unchanged public seam if table scale becomes a measured problem.
