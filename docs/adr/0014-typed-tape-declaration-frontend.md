# Build CupidC declarations directly from the typed preprocessing tape

The body grammar publishes compatible structure and union values through plain assignment, function return, automatic expression initialization, fixed arguments, and same-record conditional expressions. These paths reuse the existing assignment, call, conditional, implicit-conversion, block-binding, and return records. Aggregate lvalues receive ordinary lvalue conversion before their value flows into another context. Every converted aggregate source and destination must be a complete object type. A conditional aggregate result is not an lvalue. Plain assignment also applies C11's full modifiable-lvalue rule: the target cannot contain a direct or recursively nested `const` member or array element. An iterative type walk with visited logical IDs and a `types + members + 1` work bound checks that rule without following pointers or revisiting shared subgraphs. Aggregate initialization from one compatible expression is represented; initializer lists remain deferred.

The `aggregate-values` contract covers structures, unions, calls, chained assignment, volatile destinations, fixed arguments, return and initialization of records with `const` members, exact incompatible-type diagnostics, incomplete call and conditional operands, direct and nested nonmodifiable records, const flexible-array members, and records whose members point to const objects. It also pins non-lvalue assignment and conditional results, constrained output, rollback, tape immutability, prior-unit survival, same-job recovery, and visited-type scratch rewind. The rewind guard parses a 240-link assignment chain after introducing 600 distinct pointer types under a measured 128 KiB output ceiling; disabling the rewind reaches the public storage-limit diagnostic inside that chain. The unchanged hosted Toolchain gate now completes `cupidobj.c` with 14 definitions, 289 statements, and 1,984 expressions. `ctool.c` reaches `switch` at `122:3`, `cupiddis.c` reaches `do` at `212:3`, `cupidld.c` reaches a static local initializer at `1107:3`, and `cupidc_type.c` reaches `switch` at `309:3`.

The body grammar publishes typed C11 `while` statements for the represented scalar-expression subset. `CTOOL_C_STATEMENT_WHILE` retains one required controlling expression after ordinary lvalue, array, or function conversion and one required body statement. The body precedes its loop node in statement postorder. Integer and pointer conditions are represented; floating conditions remain a precise deferred boundary, while aggregate and `void` conditions fail their C constraints. The parser enters both breakable and iteration contexts only while it parses the body, so nested `break` and `continue` statements stay targetless for later lowering without leaking their validity outside the loop.

The public contract covers integer, pointer, volatile lvalue, array, function, and conditional-expression controls; null and compound bodies; nested loops; and loop-jump context. Nine exact failures cover aggregate, `void`, and floating controls, a direct declaration body, a missing opening or closing parenthesis, an absent condition, and absent bodies before a closing brace or at end of file. The same contract reaches the 256-level syntax limit and a constrained-output failure while proving zero failed output, exact arena and tape rollback, earlier-result survival, and same-job recovery.

The body grammar now publishes right-associative C11 conditional expressions. `CTOOL_C_EXPRESSION_CONDITIONAL` owns three ordered postorder children: the condition after ordinary conversion, the nonzero arm, and the zero arm. The node never yields an lvalue. Integer arms use the Cupid i386 promotions and usual arithmetic conversions. Compatible pointer arms use a composite referent and the union of immediate referent qualifiers. Object/`void` pairs use an appropriately qualified `void` referent and retain explicit pointer conversions on the object arm. Array and function operands decay before this selection, compatible function pointers remain valid, and two `void` arms produce `void`. Matching structure or union arms retain their common record type. Cupid follows N2209 for atomic referents in this operator. Atomic-qualified `void`, floating arithmetic arms, and unresolved null-pointer forms remain precise deferred boundaries.

The integer constant-expression grammar uses the same conditional precedence and right association in static assertions, enumerators, array bounds, bit-field widths, and alignment arguments. It parses and checks both arms, applies the common target integer kind, and suppresses divide, remainder, overflow, and shift faults only in the unselected arm. A selected fault still fails at its operator. Comma expressions remain outside both expression grammars.

The body grammar now publishes typed pointer equality and relational comparisons. The ordinary lvalue, array, and function conversions run before constraint checking, and every comparison produces target signed `int`. Equality accepts pointers to compatible types, including function and incomplete object types. It also accepts an object pointer paired with a qualified or unqualified `void` pointer and records the required object-to-void conversion as `CTOOL_C_CONVERSION_POINTER`. Relational comparison accepts compatible object pointer types. In C11 an object type may be complete, incomplete, or `void`; function types remain outside that rule. Cupid follows [WG14 N2209's proposed correction](https://www.open-std.org/jtc1/sc22/wg14/www/docs/n2209.htm): comparison and subtraction accept atomic, qualified, or unqualified versions of compatible referents, including mixed atomic pairs.

The shared assignment-conversion seam uses the same pointer conversion kind for object pointers converted to or from qualified or unqualified `void` pointers. The destination referent must retain every qualifier present on the source referent, and N2209's assignment rule requires both referents to be atomic or neither to be. Atomic-qualified `void` remains outside ADR 0013's layout contract, so the valid both-atomic object/void form is a precise unsupported type boundary. Calls, returns, automatic initializers, and assignment expressions share one checked scalar or compatible-aggregate rule instead of reconstructing it later. Pointer/integer equality remains unsupported when the integer operand could still be a zero-valued integer constant expression. Direct nonzero constants, lvalue-converted objects, calls, assignments, and updates are known not to qualify and receive input diagnostics. An iterative AST walk finds these definite disqualifiers under supported arithmetic and casts. It walks the always-evaluated left side of a logical operation but leaves the potentially skipped right side unresolved because C permits forbidden operators in unevaluated subexpressions. A possible `(void *)` cast of an unresolved zero-valued integer constant expression is also deferred before object/void handling because C defines that cast as a null pointer constant. A source literal `0` remains part of the same deferred work instead of receiving a special case. Pointer/integer relational comparison is always a constraint error.

The preprocessing tape remains the immutable lexical authority and preserves the source spelling of C11 bracket digraphs `<:` and `:>`. The frontend interprets those spellings through one canonical bracket comparison wherever its supported grammar expects `[` or `]`, including array declarators, subscripts, and the fail-closed `__builtin_offsetof` array-designator boundary. Canonical and digraph delimiters may be mixed without rewriting or mutating the borrowed tape.

Compound/update validation distinguishes represented C constraints from unimplemented arithmetic. Floating `*=`, `/=`, `+=`, and `-=` computation and floating prefix/postfix updates are valid forms but remain explicit unsupported boundaries until floating conversions and arithmetic AST exist. Floating `%=`/shift/bitwise compounds and aggregate compound/update operands instead receive input diagnostics because they violate C operand constraints. Compatible aggregate plain assignment publishes an ordinary assignment node with one raw designator and an unqualified record result.

CupidC declarations will live behind one freestanding `ctool_c_parse` operation. The operation borrows ADR 0012's immutable preprocessing-token tape plus an explicit C/Cupid language-mode request and publishes one immutable, job-owned translation-unit result. That result contains the ADR 0013 indexed type/member/parameter graph, the corresponding completed i386 layouts, canonical ordinary bindings, named tags, parameter metadata, source-ordered block bindings, and the initial immutable typed function-body AST tables. File bindings distinguish typedefs, objects, functions, and enumerators; the current block-binding slice publishes complete automatic objects and retains either their converted scalar or compatible aggregate initializer expression or `CTOOL_C_AST_NONE`. A canonical file binding retains the first declaration's storage spelling and dual location while its linkage and type describe the merged entity; a composite type is symbol-local and never mutates a shared typedef graph. Enumerator bindings retain their target-width integer bit pattern and evaluated signedness. Every public name and both its presumed and immutable physical source location are copied into the job arena; the caller's tape and request are borrowed only for the call.

This is a typed-tape frontend, not a text adapter around the private kernel compiler. It consumes the typed tape as the only lexical authority, including already-expanded spellings, source provenance, and per-token `#pragma pack` caps. A successful call invokes type layout before publishing the translation unit. Any parser, semantic, storage, limit, or layout failure zeros the result, preserves a structured diagnostic, and rewinds every allocation made since the operation mark. The input tape is never mutated. A same-job recovery contract parses a valid declaration immediately after a precise syntax failure without invalidating the earlier preprocessing result. A separate ownership contract builds a tape in caller storage, parses it, destroys and overwrites every borrowed token spelling and path, and then rereads file and block bindings, tags, members, parameters, definitions, statements, initializer roots, and expression names/locations plus decoded literal bytes from the published unit.

The frontend owns the declaration-specifier subset contracted by this slice, declarator precedence, pointers, fixed and unspecified arrays, function prototypes, parameter array/function adjustment, records, unions, Cupid classes, enums, typedef lookup, the represented storage classes and qualifiers, bit fields, and their declaration-time language rules. C11 `inline` is accepted only for declarations of function identifiers, repeated use is idempotent, non-function placements fail transactionally at the specifier, and compatible declarations OR one `CTOOL_C_FUNCTION_DECL_INLINE` summary bit into the canonical binding without changing its established C linkage. That summary is deliberately not definition policy: each function-definition record now retains its own declared type, `inline`, and storage spelling before external-inline finalization or emission. `_Thread_local`, `_Noreturn`, `_Alignas`, `_Atomic(type-name)`, and complex/imaginary type specifiers remain explicit pending syntax rather than being implied by that ownership claim. Its namespace contract keeps ordinary identifiers and tags separate, gives function prototypes their required tag/enumerator scope, rejects duplicate parameter names, and does not leak prototype-scope definitions into file scope. It rejects reserved C words in identifier positions and additionally reserves Cupid words only in Cupid mode, so `class` remains an ordinary identifier in C mode. Published file-scope binding metadata distinguishes no storage class, typedef, extern, and static plus none/internal/external linkage. Visible `static` followed by `extern`, and a static function followed by an unadorned declaration, retain internal linkage; an external declaration followed by `static`, or a static object followed by an unadorned external-linkage declaration, conflicts. File-scope auto/register are rejected; valid prototype/definition parameter `register` syntax is checked and retained in `ctool_c_parameter_t`.

Compatible object and function declarations use one iterative, memoized type-relation operation. It collapses duplicate/equivalent qualification wrappers, keeps referent qualifiers distinct, pushes array-typedef qualification to the element type, removes top-level parameter `const`/`volatile`/`restrict` while preserving atomic type identity, and treats a same-scope typedef redefinition more strictly than an object/function compatibility check. Old-style function declarations compose with prototypes only when every prototype parameter survives default argument promotion and no ellipsis is present. Incomplete and fixed arrays compose to the fixed form; opposing refinements in separate function parameters create a new composite parameter slice. Canonical binding replacement is copy-on-write: `typedef int A[]; extern A x; extern int x[4]; extern A y;` completes only `x`, leaving `A` and `y` incomplete. Repeated-pair work is memoized and stored in checked buffers, so a 24-level two-parameter compatibility DAG stays bounded, while 512 iteratively parsed pointer levels prove the 256 source-syntax limit is not misapplied to derived type graphs.

The GNU declaration subset parses normalized `__attribute` and `__attribute__` lists at active record, member, typedef, object/function, prefix, and interposed placements. `packed` updates the record/member layout policy; record alignment raises record layout; member alignment raises member placement; a typedef alignment creates an exact `CTOOL_C_TYPE_ALIGNED` wrapper without mutating its record; and object/function alignment publishes a minimum entity alignment on the canonical binding. `noreturn` is a binding-level function effect, not a type-compatibility distinction, and compatible declarations merge it in either order. Aligned wrappers compare transparently as C types while asymmetric composite selection retains the strongest alignment recursively and repeated typedefs retain the maximum. When alignment and structural refinements occur at opposing wrapper depths, composition separates common outer qualifiers from the structural children, combines the children, and reapplies exact alignment and qualifiers in source-semantic order. Contracts pin incomplete-aligned/fixed-array composition, nested exact wrappers, top-level `const`, and `_Atomic` minimum alignment in both declaration orders. Default-promotion checks unwrap alignment, so an aligned `char` prototype still conflicts with an old-style declaration in either order. Cupid i386 assigns bare `aligned`/`aligned()` the target default 16 and deterministically takes the maximum of repeated alignment requests. Unknown attributes, disabled GNU mode, malformed groups, invalid alignment, wrong arity, wrong targets, and currently unrepresented parameter attributes receive transactional `CTOOL_C_PARSE_DIAG_ATTRIBUTE` failures rather than being skipped. Unchanged source contracts pin `process_t` at `656/16` with `fp_state` at byte 80, the two packed IDT records, all five packed loader records, the per-CPU body at `128/4` with its typedef at `128/64`, both packed/aligned E1000 descriptors, and the three panic functions' retained `noreturn` effect.

Declaration-time completeness and qualification are checked where C requires them rather than inferred from final graph completeness. Every syntactic array declarator requires a complete object element type when the array is constructed, before a typedef can publish it or a parameter array can adjust to a pointer; completing a referenced tag later cannot repair an invalid array. In strict C mode an array element also cannot recursively contain a flexible-array member. By-value record members have the same temporal completeness rule, while an incomplete type remains valid in a function prototype and an external array with a complete element and unspecified bound retains its incomplete array representation. Void objects, arrays, and members are rejected; the single unnamed `void` prototype parameter is normalized to an empty prototype; named or non-sole void parameters are rejected. Array parameters adjust to pointers while retaining element qualification, and abstract `int ()`/`int (void)` parameter types adjust to pointer-to-function without manufacturing names. Qualified array objects retain their array representation, and declarations that would return arrays or functions are rejected. Flexible-array eligibility counts both direct and anonymously promoted names, accepts a promoted-name-plus-final-flexible-array record, rejects a sole flexible member, rejects an anonymous enum as a member, diagnoses promoted-name collisions, and propagates the strict-C embedding restriction through containing unions without rejecting the permitted union construction itself.

The operation now publishes the first body layer under ADR 0003 without adding a second parse seam. `ctool_c_translation_unit_t` carries immutable flat function-definition, block-binding, statement, expression, statement-child, and expression-child tables beside the declaration graph. Function definitions name their canonical binding but separately retain the exact definition declarator type, source storage, `inline` flags, body, and dual location. Definition parameter expressions index the definition type's existing parameter slice. Each `ctool_c_parameter_t` retains `register`/none storage plus the adjusted definition-local object type, including top-level qualification; the parallel `graph.parameter_types` entry remains the unqualified normalized type used for function compatibility. This split prevents a `const` or `volatile` definition parameter from becoming modifiable or losing its load semantics merely because C ignores top-level parameter qualification when comparing function types. A declaration statement indexes its ordered slice of `ctool_c_block_binding_t`; each binding owns its object name, source storage spelling, semantic type, optional converted initializer expression, and dual location. An absent initializer uses `CTOOL_C_AST_NONE`, while a block-binding expression indexes that table directly. Statement and expression nodes are postorder: every child index is smaller than its parent, slices preserve source order, and inapplicable indices use `CTOOL_C_AST_NONE`. Closed lexical scopes are parser scratch rather than public state, so their source-ordered bindings survive for later IR without exposing scope stacks, token pointers, or recursive ownership.

The body grammar deliberately follows unchanged active cohorts. It accepts nested compound, declaration, expression, return, `if`/`else`, counted `for`, typed `while`, `break`, and `continue` statements plus complete automatic-object declarations with none, `auto`, or `register` storage, comma-separated declarators, pointers, types already expressible through the supported declaration subset, and an optional scalar or compatible aggregate expression initializer for each declarator. One scalar assignment-expression may be wrapped in nested braces, and a trailing comma is accepted; empty or multi-element scalar lists fail precisely. Aggregate initializer lists remain deferred. C's point of declaration is preserved by making a provisional binding at its future stable index visible immediately after the declarator and throughout its own initializer, then publishing the finalized binding in source order. A later comma declarator is not visible early. The provisional binding retains source storage and type facts, so self-reference already uses the final index and taking the address of a pending `register` object remains invalid. The function body's outer compound shares the parameter scope: redeclaring a parameter there is rejected, while a nested block may shadow it. Lookup follows C's ordinary-identifier scopes from innermost block binding to definition parameter to file binding; leaving a compound restores the hidden outer local, parameter, file object, or typedef. Body expressions include file bindings, definition parameters, block bindings, concatenated ordinary narrow string literals, target-typed integer and ordinary narrow character constants, parentheses, fixed-argument prototyped calls, explicit scalar/void casts, unary `+`, `-`, `~`, `!`, address-of, and dereference, prefix/postfix increment/decrement, direct and anonymously promoted `.`/`->` member designators, subscripting, multiplicative/additive/shift/relational/equality/bitwise/logical binary operators, right-associative simple or compound assignment, and matching-record conditional values. Integer and character escapes use source bytes plus simple/octal/hexadecimal forms; an ordinary character byte with its high bit set is sign-extended to target signed `int` under Cupid i386's signed plain-`char` rule. Decoded strings own one trailing null and receive fixed `char[N]` types. A return statement stores either `CTOOL_C_AST_NONE` for valid `return;` from `void` or a converted scalar or compatible aggregate expression for a non-void result. An `IF` statement stores a required scalar condition after the ordinary lvalue, array, or function conversion, one required body, and an optional `else_body`; an absent alternative uses `CTOOL_C_AST_NONE`, both bodies precede their parent, and recursive statement parsing gives a dangling `else` to the nearest unmatched `if`. Selection adds no scope or loop-jump target of its own. Calls enforce prototype arity and accept compatible scalar or aggregate arguments plus integer and pointer-qualification assignment conversion. A variadic prototype may be called with exactly its named arguments, but an argument in the ellipsis fails closed until integer and floating default argument promotions have explicit AST representation. Valid decimal and hexadecimal floating constants are recognized as such and receive an explicit unsupported-expression diagnostic rather than falling through to a malformed-integer diagnostic. A floating source or target at the shared assignment-conversion seam and a floating operand to `!`, `&&`, or `||` receive distinct unsupported diagnostics until floating arithmetic, numeric conversion, and ABI lowering exist.

The first counted-control layer represents C11 `for` without inventing a control-flow graph prematurely. `CTOOL_C_STATEMENT_FOR` retains initializer, condition, iteration, and body fields. The initializer field is `CTOOL_C_AST_NONE` when omitted; when present, it names either the existing non-null expression-statement form or the existing declaration-statement form. The condition and iteration are optional, and the body is required. A missing condition likewise uses `CTOOL_C_AST_NONE` and denotes C's nonzero constant, while a present condition receives lvalue, array, or function conversion and must be scalar. Integer and pointer conditions are represented now; a floating condition is valid C but receives the precise deferred-feature diagnostic until floating conversions exist, whereas aggregate and `void` conditions are constraint failures. A present iteration expression receives the ordinary default conversion. `BREAK` and `CONTINUE` are targetless postorder leaves; their nearest enclosing loop or future switch target belongs to lowering, while the frontend rejects either outside its permitted enclosing control context. The initializer and body precede their `FOR` parent, and every expression remains postorder in the existing expression table.

C11's loop scope is explicit. A `for` declaration initializer accepts only complete automatic objects with none, `auto`, or `register` storage under the existing declaration constraints; `typedef`, `extern`, `static`, function declarations/specifiers, and other nonautomatic forms fail at the initializer rather than being silently downgraded. The scope covers the declaration initializer, condition, iteration, and body, permits an initializer to shadow a function parameter, and expires immediately after the loop. Parsing now distinguishes a compound statement's block items from the statement grammar: declarations remain valid block items and declaration initializers, but a declaration cannot serve directly as the loop body without a compound statement. This split also makes standalone and loop-body `;` publish a null expression statement. Comma expressions remain outside the body-expression subset; comma-separated declaration declarators remain supported, but an expression initializer such as `i = 0, i = 1` reaches the explicit expression boundary.

Semantic conversions remain explicit one-child AST nodes rather than facts that IR must reconstruct. Lvalue-to-value, array-to-pointer, function-to-pointer, qualification, integer-promotion, usual-arithmetic, and assignment conversions each publish their result type while preserving the source expression as a child. Lvalue conversion removes top-level `const`, `volatile`, and `_Atomic` from the result while the child retains the qualified object type needed to select the load. Array decay transfers array qualification to the pointed-to element, including qualification inherited from a containing record. Scalar initialization applies that lvalue conversion to the declared object type and then reuses the shared assignment-conversion operation, but does not apply assignment's modifiable-lvalue check because a declaration is not an assignment expression. The binding therefore retains its qualified or atomic object type while its initializer root is the converted, unqualified value expression. Plain assignment retains its raw modifiable-lvalue child, converts its right operand to the left object type, and produces an unqualified result; its `computation_type` equals that result type, an invariant rechecked while freezing the public graph. Compound assignments retain the raw left designator exactly once and record the promoted/usual-arithmetic computation type separately from the unqualified stored/result type. Shift assignments use the promoted left type without inventing a usual-arithmetic conversion, and pointer `+=`/`-=` use the pointer result as their computation type plus a promoted integer right child. Prefix/postfix updates likewise publish one raw designator child, distinct operations, the stored/result type, and the promoted integer or pointer computation type. Compatible enum/integer assignment still publishes an explicit assignment conversion to the enum destination instead of disappearing through the compatibility relation. Integer operators use explicit Cupid i386 ranks `_Bool < char < short < int < long < long long`, with signed plain `char`, 32-bit `int` and `long`, and 64-bit `long long`. Integer promotions and the usual arithmetic conversions are represented in the AST, so the ILP32 case `long + unsigned int` resolves to `unsigned long` instead of inheriting a host-compiler rank or width. A narrow unsigned-`int` bit-field whose values fit target signed `int` publishes lvalue conversion to unsigned `int` followed by integer promotion to signed `int`; a 32-bit field remains unsigned. Comparisons and logical operators produce target signed `int`. The callee and converted arguments are the ordered call children, and the call type is the function result.

`CTOOL_C_EXPRESSION_CAST` retains the explicitly requested scalar or `void` destination type instead of disguising source casts as implicit conversions. Its child still undergoes the ordinary lvalue, array, or function conversion, including for a `void` discard, so volatile accesses and decay cannot disappear. Address-of and dereference are unary nodes with their source-semantic child, so later lowering can distinguish an address computation from a load. Addressability tracks the C exclusions for bit-fields and `register` objects through every member hop, while member lvalues inherit the effective record and member qualification. Every `CTOOL_C_EXPRESSION_MEMBER` names one direct member index in the ADR 0013 graph. Lookup through an anonymous structure or union resolves a promoted name to an ordered path and publishes one member node per direct hop; no synthetic flattened member or pre-summed offset is introduced into the public AST.

Pointer addition accepts pointer-plus-integer in either source order, pointer subtraction accepts an integer right operand, and compatible pointer subtraction produces target signed `int`, the Cupid i386 `ptrdiff_t` representation. Every participating pointer must refer to a complete object type; `void`, function, and incomplete referents fail precisely. Pointer difference ignores pointed-to `const`, `volatile`, and `restrict` while retaining atomic type identity. A subscript is normalized to one pointer-addition node followed by dereference, matching C's `*((a) + (b))` definition while retaining operand order, target integer promotion, element qualification, lvalue state, and one lowering path. Pointer comparison is represented; implicit null-pointer-constant conversion remains pending.

Ordinary body expressions are type-checked but deliberately not evaluated by the integer-constant-expression engine. Source such as `1 / 0`, `2147483647 + 1`, or `1 << 32` remains a typed runtime expression rather than receiving a translation-time arithmetic diagnostic or being folded away. The deliberate exception is a non-VLA layout query: `sizeof`, alignment queries, and `__builtin_offsetof` become target unsigned 32-bit integer constants after their operands or member paths have been checked against the current Cupid i386 graph. GNU expression alignment uses the selected final member declaration; neither a packed anonymous-promotion container, a named packed container, nor an aligned containing object changes that member's reported alignment. This is distinct from enumerator, bound, bit-field, and `_Static_assert` expressions, where the existing evaluator must diagnose selected arithmetic faults and suppress only faults in a skipped short-circuit operand. Keeping the policies separate preserves runtime source while still sharing target types, conversions, member lookup, and layout.

The first full-precedence body parser mirrored the grammar with one recursive wrapper function per binary precedence tier. Re-entering that entire wrapper chain for each nested call multiplied host-stack use enough that the existing 256-deep call contract exhausted the default Windows stack before it could emit the public syntax-limit diagnostic. The final binary grammar uses checked value and operator vectors plus an iterative precedence reducer. Unary nesting, calls, parentheses, and right-associative assignment still consume the explicit syntax budget, while ordinary binary chains add no precedence-wrapper recursion. This preserves left associativity and postorder AST publication and keeps the advertised 256-level nested-call guard reachable on the supported Windows host. A 4,096-operator flat addition contract publishes exactly 8,193 postorder expressions and 8,192 child references, then walks the complete left-associated chain. Re-running that source with a 256-byte output limit produces exactly one public limit diagnostic, zeros the failed unit, restores the arena and scratch allocator balance, preserves the tape and an earlier anchor unit, and permits same-job recovery.
