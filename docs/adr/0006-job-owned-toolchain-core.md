# Put deterministic tool state behind a job-owned core

Cupid Toolchain modules will share one freestanding core. Its opaque job owns an arena, logical paths, loaded sources, and structured diagnostics. Checked buffers remain explicit bounded handles, and invocation output belongs to one job. Platform variation is limited to narrow allocator, whole-file size/read/write, and text-sink adapters. Hosted adapters may use the C runtime; the kernel adapter uses the Cupid OS heap, VFS, and output path. Core paths are absolute logical `/` paths. Core sizes and serialized offsets are checked 32-bit values, diagnostics retain emission order, and an invocation commits output only when its body succeeds without an error diagnostic. Diagnostics use bounded job-owned slots outside the rewindable operation arena, so a failed transactional operation can reclaim its scratch storage without invalidating the error report.

Tool-specific language, x86, ELF, linker, symbol-catalogue, and JIT policy remain typed modules above this seam. Kernel bindings, executable-memory policy, shell state, and stack guards do not become platform callbacks. A single giant platform vtable was rejected because it would relocate the current coupling instead of hiding it; bare allocation/file callbacks without job ownership were rejected because each tool would repeat lifetime, limit, path, output, and diagnostic policy.

The core and both real adapters are buildable. CupidASM and CupidDis share their freestanding implementations between native CLIs and kernel adapters; CupidObj and CupidLD have native CLIs over the same job seam. CupidC preprocessing, declaration parsing, i386 type layout, typed leaf IR, and mixed static/function object emission now have hosted freestanding operations over the same job contract. All hosted tools are still compiled by the host C compiler, and the host C compiler still produces the normal root and user C objects. The private kernel CupidC frontend remains the embedded runtime JIT and AOT path. The remaining AST and IR surface, calls and relocations, kernel integration, and a host-runnable compiler are still required before staged self-hosting.
