# Keep CupidC type layout behind one immutable indexed operation

CupidC type representation and i386 object layout will live behind one freestanding `ctool_c_layout_types` operation. A request borrows immutable, flat arrays of type nodes, record members, and function-parameter type indices. References are indices into those arrays; a successful call publishes parallel type/member layout arrays in the same order, owned by the Toolchain job arena. A failure publishes one structured diagnostic, zeros the result, and rewinds every allocation made by the operation. Temporary visit, ownership, and traversal arrays are allocated after a scratch mark and reclaimed before success without invalidating any previously published result.

This is the first semantic layer under ADR 0003, not a replacement parser. It preserves distinct C scalar identities even when their representations match and fixes layout to the Cupid i386 target instead of asking the compiler hosting the bootstrap. The contract includes signed plain `char`; one-byte `_Bool` and character types; two-byte `short`; four-byte `int`, `long`, enums, pointers, and `float`; eight-byte `long long` and `double` with four-byte ordinary i386 alignment; the active 16-byte vector forms; fixed and unspecified arrays; function and function-pointer markers; and complete/incomplete struct, union, and Cupid class records. Atomic scalar, pointer, and enum representation is explicit target policy: atomic `long long` and `double` raise alignment to eight while other supported scalar cases retain their fixed size/alignment. Atomic aggregate representation is deliberately unsupported until an active ABI requirement establishes it.

By-value dependencies are strong graph edges. The operation lays them out with an iterative color/stack walk, rejects a strong cycle at the responsible presumed source location, and permits weak pointer recursion. This avoids using the host call stack for adversarial type graphs. Every count, slice, reference, multiplication, alignment, and final size is checked before use. Variable-array layout is reserved but unsupported; flexible arrays are represented by an unspecified-bound array, and layout enforces only that such a member is final and belongs to a structure. The declaration frontend owns the C requirement that the structure have more than one named member, including names promoted through anonymous records. It also owns function return/parameter constraints, qualifier placement, duplicate member names, namespaces, and whether anonymous-member syntax is legal in the selected language mode.

Record layout consumes preprocessing metadata without replaying source text. Each member records the effective `#pragma pack` cap from the first declaration-specifier token; all declarators in that member declaration inherit that sample. Record/member `packed`, member alignment, the per-member pack cap, and final record alignment are applied explicitly. An alignment-carrying type wrapper records the exact effective alignment of an aligned typedef or type attribute without mutating the wrapped type definition. It may lower or raise natural alignment: the positive scalar oracle wraps a four-aligned `int` with effective alignment one, while the active `per_cpu_t` wrapper raises a four-aligned record to 64. A record definition's own explicit alignment remains different and can only raise its computed alignment.

The per-member pack model is a deliberate deterministic choice. A local GCC i386 probe for a `pack(push, 1)` member followed by `pack(pop)` before the next member produced size/alignment `8/4` and placed the second member at byte 4; Clang instead retained a record-wide snapshot and produced `5/1` with the member at byte 1. Cupid follows the GCC result because the typed preprocessing tape already stamps the declaration token at the point where the member begins. Further GCC/Clang probes established the contracted interaction: a packed record may have an explicitly aligned member raise its effective alignment, while an active pragma cap limits that member alignment; explicit record alignment raises the final record alignment. Bit-field probes established least-significant-bit allocation within the current integer storage unit, zero-width boundary behavior, and the active Doom color layout at bit offsets 0, 8, 16, and 24.

The hosted contract manually constructs graphs for active ABI shapes. It checks all 54 members from unchanged `kernel/fs/fat16.h` as semantic oracles: packed partition/MBR/boot-sector/directory-entry records are respectively `16/1`, `512/1`, `36/1`, and `32/1`, while the runtime filesystem/file records are `36/4` and `24/4`, with every member offset pinned. It also checks the Doom four-channel bit field, nested and anonymous records, unions, final flexible arrays, packed and aligned descriptors, the exact one-aligned `int` wrapper, a 656-byte/16-aligned process shape, the 103-pointer 412-byte syscall table, and the 128-byte `per_cpu_t` body plus its 64-aligned typedef wrapper. These are manual typed-graph layout oracles; CupidC does not yet parse those unchanged headers into the graph, so they transfer no production compilation ownership.

Adapting the private kernel parser was rejected. Its text re-lexing loses counted spellings, physical/presumed locations, token columns, per-token pack state, and important type distinctions, and it emits machine code while parsing. Making that interface public would preserve the wrong seam and obstruct the shared typed AST and linear IR. The next slice is therefore a declaration frontend that consumes the immutable preprocessing tape directly, constructs this semantic graph, and parses the unchanged FAT16 header before kernel integration is attempted.

This capability is hosted bootstrap evidence only. The host compiler still builds the module and its contract; the private in-kernel CupidC frontend does not consume it; no OS artifact changed; no boot behavior is claimed; and `TempleOS/` remains excluded reference material.
