# Keep CupidC type layout behind one immutable indexed operation

CupidC type representation and i386 object layout will live behind one freestanding `ctool_c_layout_types` operation. A request borrows immutable, flat arrays of type nodes, record members, and function-parameter type indices. References are indices into those arrays; a successful call publishes parallel type/member layout arrays in the same order, owned by the Toolchain job arena. A failure publishes one structured diagnostic, zeros the result, and rewinds every allocation made by the operation. Temporary visit, ownership, and traversal arrays are allocated after a scratch mark and reclaimed before success without invalidating any previously published result.

This is the first semantic layer under ADR 0003, not a replacement parser. It preserves distinct C scalar identities even when their representations match and fixes layout to the Cupid i386 target instead of asking the compiler hosting the bootstrap. The contract includes signed plain `char`; one-byte `_Bool` and character types; two-byte `short`; four-byte `int`, `long`, pointers, and `float`; eight-byte `long long` and `double` with four-byte ordinary i386 alignment; the active 16-byte vector forms; fixed and unspecified arrays; function and function-pointer markers; layout-preserving qualified-type wrappers; and complete/incomplete struct, union, and Cupid class records. An aligned incomplete object retains size zero, incomplete-object state, and its exact known alignment; a non-object target remains invalid. This lets a later compatible declaration combine an aligned unspecified array with its fixed bound without discarding either fact. An enum node references the compatible integer type selected by the frontend and copies that type's size, alignment, and signedness. The manual active-source layout fixtures select signed `int` and are therefore `4/4`; ADR 0014 now makes source-driven choices, selecting unsigned `int` for the two nonnegative enums in the FAT16 closure and signed `int` or signed `long long` for contracted mixed-value cases. Atomic scalar, pointer, and enum representation is explicit target policy. Positive contracts pin a direct atomic pointer at `4/4` and a synthetic signed-`long long`-compatible atomic enum at `8/8`. Introducing `_Atomic` on `long long` or `double` likewise raises alignment to eight while other supported scalar cases retain their target minimum. Introduction is ordered with alignment attributes. An `ALIGNED` or `QUALIFIED` node that introduces `_Atomic` uses the greater of its current/exact alignment and the target minimum. A later `ALIGNED` node that merely wraps an already-atomic type applies its exact alignment and may lower it; a later `QUALIFIED` node adding only `const`, `volatile`, or `restrict` copies that effective layout unchanged. The GCC/Clang oracle is aligned `int` `1 -> atomic 4 -> realigned 1`, and aligned `long long` `1 -> atomic 8 -> realigned 1`. Atomic aggregate representation introduced through either wrapper is deliberately unsupported until an active ABI requirement establishes it.

By-value dependencies are strong graph edges. The operation lays them out with an iterative color/stack walk, rejects a strong cycle at the responsible presumed source location, and permits weak pointer recursion. This avoids using the host call stack for adversarial type graphs. During that single traversal it caches `_Bool` identity, active atomic state, and target atomic minimum alignment once per type, propagating all three facts through aligned and qualified wrappers. Bit-field and wrapper layout then reads those facts in constant time instead of repeatedly unwrapping derived types, keeping the operation `O(types + members)`. Every count, slice, reference, multiplication, alignment, and final size is checked before use. Variable-array layout is reserved but unsupported; flexible arrays are represented by an unspecified-bound array, and layout enforces only that such a member is final and belongs to a structure. The declaration frontend owns the C requirement that the structure have more than one named member, including names promoted through anonymous records. It also owns function return/parameter constraints, qualifier placement, duplicate member names, namespaces, and whether anonymous-member syntax is legal in the selected language mode.

Record layout consumes preprocessing metadata without replaying source text. Each member records the effective `#pragma pack` cap from the first declaration-specifier token; all declarators in that member declaration inherit that sample. Record/member `packed`, member alignment, the per-member pack cap, and final record alignment are applied explicitly. An alignment-carrying type wrapper records the exact effective alignment of an aligned typedef or type attribute without mutating the wrapped type definition. It may lower or raise natural alignment: the positive scalar oracle wraps a four-aligned `int` with effective alignment one, while the active `per_cpu_t` wrapper raises a four-aligned record to 64. A record definition's own explicit alignment remains different and can only raise its computed alignment.

`CTOOL_C_TYPE_QUALIFIED` is the separate semantic wrapper for `const`, `volatile`, `restrict`, and `_Atomic`. It copies the wrapped type's size, alignment, completeness, object, integer, and signedness facts rather than defining another record or owning another member slice. This lets the frontend represent both an unqualified nominal `T` and `const T` in one immutable graph without duplicating a record definition. It also preserves the declarator distinction between `const T *` (a pointer whose referent is `QUALIFIED(T, const)`) and `T * const` (a pointer node carrying `const`). A zero-qualifier wrapper is rejected. Qualification that does not introduce `_Atomic` is layout-preserving even when the wrapped type is already atomic. Qualification that introduces `_Atomic` raises alignment to the cached target minimum and remains unsupported for aggregate referents. Atomic identity continues through a later `ALIGNED` wrapper, but that later wrapper's exact alignment is authoritative and can lower the effective alignment again.

The per-member pack model is a deliberate deterministic choice. A local GCC i386 probe for a `pack(push, 1)` member followed by `pack(pop)` before the next member produced size/alignment `8/4` and placed the second member at byte 4; Clang instead retained a record-wide snapshot and produced `5/1` with the member at byte 1. Cupid follows the GCC result because the typed preprocessing tape already stamps the declaration token at the point where the member begins. Further GCC/Clang probes established the contracted interaction: a packed record may have an explicitly aligned member raise its effective alignment, while an active pragma cap limits that member alignment; explicit record alignment raises the final record alignment. Bit-field probes established least-significant-bit allocation within the current integer storage unit, zero-width boundary behavior, and the active Doom color layout at bit offsets 0, 8, 16, and 24.

The hosted layout contract manually constructs graphs for active ABI shapes. It checks all 54 members from unchanged `kernel/fs/fat16.h` as semantic oracles: packed partition/MBR/boot-sector/directory-entry records are respectively `16/1`, `512/1`, `36/1`, and `32/1`, while the runtime filesystem/file records are `36/4` and `24/4`, with every member offset pinned. It also checks the Doom four-channel bit field, the active `drivers/e1000.c` receive descriptor at size/alignment `16/16` and offsets 0/8/10/12/13/14, the exact one-aligned `int` wrapper, a 656-byte/16-aligned process shape, the 103-pointer 412-byte syscall table, and the 128-byte `per_cpu_t` body plus its 64-aligned typedef wrapper. Nested/anonymous records, qualified nominal records and pointer-level qualification, unions, final flexible arrays, packed/explicit-alignment composition, and atomic `QUALIFIED`/`ALIGNED` chains have separate synthetic fixtures. A dedicated adversarial fixture chains 4,096 aligned wrappers above `_Bool` and lays out 4,096 one-bit fields through the outermost wrapper twice. It pins immutable inputs, deterministic 512-byte output, correct cached facts, and the declared `O(types + members)` bound; repeated wrapper walks per member would make this fixture quadratic. The negative mode publishes 27 precise diagnostics and then proves a valid layout can recover in the same job. These remain independent manual typed-graph ABI oracles. ADR 0014 additionally constructs the same FAT graph from the unchanged typed preprocessing tape, without transferring production compilation ownership.

Final spec review found that qualifiers stored only on existing nodes could not represent both qualified and unqualified uses of one nominal record: cloning the record would either duplicate its member slice or violate unique member ownership. It would also leave pointer-level qualification ambiguous to downstream semantic consumers even though layout bytes match. The layout-preserving `QUALIFIED` node was chosen over record cloning and over discarding qualifiers after layout; it retains semantic identity without widening the record-layout interface.

Adapting the private kernel parser was rejected. Its text re-lexing loses counted spellings, physical/presumed locations, token columns, per-token pack state, and important type distinctions, and it emits machine code while parsing. Making that interface public would preserve the wrong seam and obstruct the shared typed AST and linear IR. ADR 0014 therefore consumes the immutable preprocessing tape directly, constructs this semantic graph, and parses the unchanged FAT16 header before kernel integration is attempted.

This capability is hosted bootstrap evidence only. The host compiler still builds the module and its contract; ADR 0014's hosted declaration frontend is its only source parser; the private in-kernel CupidC frontend does not consume it; no OS artifact changed; no boot behavior is claimed; and `TempleOS/` remains excluded reference material.
