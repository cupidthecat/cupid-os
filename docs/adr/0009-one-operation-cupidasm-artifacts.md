# Keep CupidASM behind one artifact-producing operation

CupidASM exposes one freestanding assembly operation: a caller supplies a Toolchain job, one source, an explicit raw, ELF32 relocatable-object, or fixed-memory-image request, immutable constants and absolute bindings when needed, and an empty output buffer; success returns committed bytes plus only the region and entry metadata required by the kernel adapter. Lexing, `%define` and include expansion, expressions, dot-local labels, sections, symbols, fixups, deterministic branch relaxation, shared-x86 lowering, and shared-ELF specification construction remain private. Raw output owns flat layout, ELF output delegates serialization to `ctool_elf32_write`, and the fixed-image profile resolves one assembly unit into caller-placed code/data regions without claiming multi-object or linker-script behavior.

A published semantic assembly-unit graph with separate assemble, layout, and emit operations was rejected for now. It would make future inspection and debug consumers convenient, but no current caller needs the assembler's internal section/fixup/source-range representation, and freezing that larger interface would reduce locality while the real source corpus is still sharpening the model. A file-only operation was also rejected because it would leave the in-kernel JIT on a second parser or force its executable-memory policy into the shared module.

The shared x86 and ELF32 modules are direct in-process dependencies. Includes use the Toolchain job's existing file-store seam, with including-file and ordered logical include-root resolution. Symbols are case-sensitive by default; an explicit request flag preserves the legacy in-kernel CupidASM's ASCII-insensitive source/binding policy without weakening NASM-compatible host assembly. Hosted filesystem and kernel VFS behavior remain platform adapters. The kernel adapter supplies its runtime constants and absolute callable symbols, copies successful fixed-image regions, owns shell/stack/executable-memory policy, and temporarily wraps the same regions in the legacy fixed-address `ET_EXEC`; final executable layout still belongs to CupidLD.
