# Share a typed AST and linear IR across two language modes

CupidC will parse C mode and Cupid mode into a shared typed AST and lower both through a shared linear IR, with C semantics and Cupid-specific extensions kept distinct at the language boundary. This replaces direct parser-to-machine-code generation while avoiding two independent compilers, enabling semantic checking and optimization without forcing Cupid extensions into ordinary C.

ADR 0013 establishes the first shared semantic layer beneath that frontend: one immutable indexed type graph and fixed i386 layout operation. The graph is parser-neutral and does not expose the private kernel parser or its direct code-generation state. ADR 0014 consumes ADR 0012's typed preprocessing tape, applies the contracted declaration and language-mode rules, constructs that graph, completes i386 layout, and publishes immutable postorder function-body tables through the same transactional operation. The unchanged FAT16 closure still reproduces the manual ABI oracles, while unchanged `debug.h` proves compound and expression statements, binding and parameter references, ordinary strings, fixed-argument prototyped calls, and explicit semantic conversions. Focused contracts also prove pointer arithmetic and subscripts; pointer equality and relational comparison; explicit object/void pointer conversion for equality and assignment; one-evaluation compound and update nodes with distinct result and computation types; typed C11 selection; and typed C11 `for` statements.

An `IF` node retains a required scalar condition after the ordinary default conversions, one required body, and an optional `else_body` that uses `CTOOL_C_AST_NONE` when absent. Both bodies precede the node in statement postorder. Recursive statement parsing binds `else` to the nearest unmatched `if`, and no control-flow target is embedded before IR lowering. A `FOR` node uses `CTOOL_C_AST_NONE` when its initializer is omitted and otherwise names a present expression or declaration statement. It also retains an optional scalar condition after the ordinary default conversions, an optional converted iteration expression, and one required body statement. Null expression statements retain `CTOOL_C_AST_NONE`, while `BREAK` and `CONTINUE` are targetless leaves whose nearest target remains a lowering decision. One loop scope covers a declaration initializer, both expression clauses, and the body, then expires. Scalar automatic-object initializers retain a converted typed expression on each block binding, honor C's point of declaration, and accept nested single-element scalar brace wrappers with an optional trailing comma. Function definitions retain their own declared type, `inline`, storage, parameters, body, and dual location separately from the canonical entity summary.

Pointer comparisons apply the ordinary lvalue, array, and function conversions. Equality accepts pointers to compatible types, plus an object pointer paired with a qualified or unqualified `void` pointer. The object operand receives an explicit `CTOOL_C_CONVERSION_POINTER` node in the latter case. Relational comparison requires compatible object pointer types and produces target signed `int`; C object types may be complete, incomplete, or `void`, while function types remain outside that constraint. Cupid follows [WG14 N2209's proposed correction](https://www.open-std.org/jtc1/sc22/wg14/www/docs/n2209.htm) for atomic referents: comparison and subtraction accept atomic, qualified, or unqualified versions of compatible types, including mixed atomic pairs. Assignment separately requires both referents to be atomic or neither to be.

The same pointer conversion kind represents assignment conversion between object and `void` pointers when the destination referent retains every source qualifier and both referents are atomic or neither is. Atomic-qualified `void` is still outside ADR 0013's layout contract, so the both-atomic object/void assignment allowed by N2209 remains unsupported. Pointer/integer equality stays deferred when the integer operand might be a zero-valued integer constant expression. Direct nonzero constants, lvalue-converted objects, calls, assignments, and updates are known not to qualify and receive constraint diagnostics. A checked iterative AST walk finds those disqualifiers beneath supported arithmetic and casts without recursing through long expression chains. For logical short-circuit operations it checks the always-evaluated left operand and leaves the potentially skipped right operand unresolved, so a call in an unevaluated operand is not falsely rejected. A possible `(void *)` cast of an unresolved zero-valued integer constant expression is likewise deferred before an incorrect object/void conversion can be published. Literal `0` remains unsupported with the other candidate constant-expression forms rather than receiving a one-token shortcut.

Floating controlling expressions are valid but deferred until floating conversions exist, whereas aggregate and `void` conditions violate the scalar constraint. Conditional and comma expressions remain deliberate boundaries. Aggregate automatic, file-scope, and static-duration initializers; the remaining control flow and expressions; floating and null-pointer-constant conversions; translation-unit finalization; linear IR; code generation; and production compiler ownership remain later gates.
