# Share a typed AST and linear IR across two language modes

CupidC will parse C mode and Cupid mode into a shared typed AST, then lower both through a shared linear IR. C semantics and Cupid-specific extensions remain distinct at the language boundary. The shared representation replaces direct parser-to-machine-code generation without creating two compilers. It permits semantic checking and optimization without adding Cupid extensions to ordinary C.

ADR 0013 establishes one immutable indexed type graph and fixed i386 layout operation beneath the shared frontend. ADR 0014 consumes ADR 0012's typed preprocessing tape, applies C or Cupid language rules, completes layout, and publishes immutable bindings, definitions, initializer forests, and typed function-body tables. ADR 0015 consumes that frozen translation unit and owns deterministic i386 ELF32 object construction for represented static data. ADR 0016 adds the first public linear IR operation and extends the object path with represented leaf functions. ADR 0017 adds fixed direct calls with source-order evaluation and PC-relative function relocations. The unchanged FAT16 closure still reproduces the manual ABI oracles, while active and focused contracts cover structured statements, parameter and object references, calls, conversions, pointer semantics, assignment and updates, conditional values, and the first 32-bit function-lowering slices.

An `IF` node retains a required scalar condition after the ordinary default conversions, one required body, and an optional `else_body` that uses `CTOOL_C_AST_NONE` when absent. Both bodies precede the node in statement postorder. Recursive statement parsing binds `else` to the nearest unmatched `if`, and no control-flow target is embedded before IR lowering. A `FOR` node uses `CTOOL_C_AST_NONE` when its initializer is omitted and otherwise names a present expression or declaration statement. It also retains an optional scalar condition after the ordinary default conversions, an optional converted iteration expression, and one required body statement. `WHILE` and `DO` nodes retain one converted scalar condition and one body. A `DO` body and its expressions precede the condition and loop node, which preserves the source's post-test order. Later IR lowering must send a `continue` in that body to the condition, not back to the body start. A `SWITCH` node retains the integer-promoted condition and its required body. Each `CASE` retains one folded integer constant converted to that promoted type, while `DEFAULT` retains only its body. Nested switch contexts isolate their values and default labels. Duplicate values are rejected after conversion, as required by [C11 draft N1570, 6.8.4.2](https://www.open-std.org/jtc1/sc22/wg14/www/docs/n1570.pdf). Null expression statements retain `CTOOL_C_AST_NONE`, while `BREAK` and `CONTINUE` are targetless leaves whose nearest target remains a lowering decision. A switch supplies a `break` context but not an iteration context. One loop scope covers a declaration initializer, both expression clauses, and the body, then expires. A block binding keeps an automatic or block-static object's identity. A separate file object definition points back to the canonical binding while retaining definition-local type, storage, kind, location, and initializer ownership. Initialized automatic scalars and whole records use converted `EXPRESSION` roots. Static objects use `ZERO`, target-converted `INTEGER`, narrow character-array `STRING`, string `ADDRESS`, an `ADDRESS` of a linked file object or function, or recursive `LIST` records. Explicit static null pointer constants use destination-typed `ZERO` records, including real children in aggregate lists. Tentative file definitions receive a zero root during translation-unit finalization. Omitted subobjects have no list edge and remain implicit zero. Other list edges name direct array elements or structure members in postorder. A direct C11 designator selects one immediate subobject, preserves source order, and moves the following positional cursor. Unknown-bound arrays receive private completed types sized from the greatest selected index. Duration-aware validation accepts `ZERO` only in static forests. Chained designator paths, promoted anonymous-member designators, duplicate overrides, and union or Cupid class lists remain deferred. Function definitions retain their own declared type, `inline`, storage, parameters, body, and dual location separately from the canonical entity summary.

Identifier labels use their C11 function scope and a namespace separate from ordinary identifiers. Each definition owns a contiguous slice of canonical label records. The first definition or `goto` creates the identity, and the label definition later fills its source locations and defining `LABEL` statement. A `LABEL` node owns its body as an ordinary postorder child. A `GOTO` node is a leaf with a semantic reference to the canonical label. That reference may point forward or backward and is not an AST child. Duplicate definitions fail immediately, and unresolved targets fail when the function ends. Computed goto and GNU label addresses remain explicit extension boundaries. The C11 rule against jumping into a variably modified object's scope needs no separate check yet because variable-length arrays are still unsupported.

Pointer comparisons apply the ordinary lvalue, array, and function conversions. Equality accepts pointers to compatible types, an object pointer paired with a qualified or unqualified `void` pointer, and a pointer paired with a represented null pointer constant. The object operand receives an explicit `CTOOL_C_CONVERSION_POINTER` node in the object/void case. A null operand receives a destination-typed `CTOOL_C_CONVERSION_NULL_POINTER` node. Relational comparison requires compatible object pointer types and produces target signed `int`; C object types may be complete, incomplete, or `void`, while function types remain outside that constraint. Cupid follows [WG14 N2209's proposed correction](https://www.open-std.org/jtc1/sc22/wg14/www/docs/n2209.htm) for atomic referents: comparison and subtraction accept atomic, qualified, or unqualified versions of compatible types, including mixed atomic pairs. Assignment separately requires both referents to be atomic or neither to be.

The same pointer conversion kind represents assignment conversion between object and `void` pointers when the destination referent retains every source qualifier and both referents are atomic or neither is. Atomic-qualified `void` is still outside ADR 0013's layout contract, so the both-atomic object/void assignment allowed by N2209 remains unsupported. Runtime expression values carry private integer-constant-expression form, value validity, and target-width value metadata. The metadata follows the represented integer operator grammar, promotions, arithmetic conversions, integer casts, short-circuit evaluation, and conditional selection without changing the public AST or folding runtime expressions. A represented integer constant expression with value zero is a null pointer constant. Casting that expression to a pointer whose referent is unqualified `void` also qualifies when the pointer itself is not atomic-qualified. Calls, assignments, updates, typed pointer casts, nonzero values, selected invalid arithmetic, and atomic-qualified casts do not qualify. This exact classification replaced the earlier iterative AST disqualifier walk and the rejected literal-only shortcut.

A `CONDITIONAL` expression owns three ordered postorder children: the converted condition, the nonzero arm, and the zero arm. It never yields an lvalue. Integer arms use the Cupid i386 usual arithmetic conversions. Compatible pointer arms build a composite referent and merge its immediate qualifiers; object/`void` pairs produce an appropriately qualified `void` referent. A pointer and null pointer constant pair produces the pointer type and a destination-typed null conversion on the other arm. N2209 atomic qualification participates in pointer merges, while atomic-qualified `void` remains the explicit ADR 0013 boundary. Array and function arms decay before result selection. Two `void` arms produce `void`. Later IR lowering must evaluate the condition, observe C11's sequence point, evaluate only the selected arm and its conversions, and join the result. Floating and compatible aggregate forms outside the represented subset remain valid but deferred. The integer constant-expression path uses the same right association and evaluates only the selected arm while still parsing both.

ADR 0016 now proves that selected-arm behavior for the represented 32-bit integer subset. Its function-relative branch targets keep the choice out of the AST, and its address/value stack has the same shape at the join.

Floating controlling expressions for loops and `if` remain valid but deferred until floating conversions exist, whereas aggregate and `void` conditions violate the scalar constraint. A switch condition must have integer type, so pointer, floating, aggregate, and `void` controls are rejected. Comma expressions remain a deliberate boundary. ADR 0015 owns static section placement, bytes, symbols, and `R_386_32` relocations. ADR 0016 owns typed leaf IR and the first `.text` emission. ADR 0017 owns fixed direct calls and their `R_386_PC32` relocations. ADR 0018 owns the first automatic 32-bit integer locals and their fixed EBP slots. Chained or overriding initializer designators, promoted anonymous-member designators, union or Cupid class lists, remaining address forms, floating values, computed goto, GNU label addresses, external-inline finalization, indirect and variadic calls, locals outside the ADR 0018 slice, call-site alignment, general control-flow lowering, the remaining AST and IR, whole-unit emission, and production compiler ownership remain later gates.
