# Share a typed AST and linear IR across two language modes

CupidC will parse C mode and Cupid mode into a shared typed AST and lower both through a shared linear IR, with C semantics and Cupid-specific extensions kept distinct at the language boundary. This replaces direct parser-to-machine-code generation while avoiding two independent compilers, enabling semantic checking and optimization without forcing Cupid extensions into ordinary C.

ADR 0013 establishes the first shared semantic layer beneath that frontend: one immutable indexed type graph and fixed i386 layout operation. The graph is parser-neutral and does not expose the private kernel parser or its direct code-generation state. ADR 0014 consumes ADR 0012's typed preprocessing tape, applies the contracted declaration and language-mode rules, constructs that graph, completes i386 layout, and publishes one semantic initializer forest with immutable initializer records and direct-subobject edges, plus immutable postorder function-body tables through the same transactional operation. The unchanged FAT16 closure still reproduces the manual ABI oracles, while unchanged `debug.h` proves compound and expression statements, binding and parameter references, ordinary strings, fixed-argument prototyped calls, and explicit semantic conversions. Focused contracts also prove pointer arithmetic and subscripts; pointer equality and relational comparison; explicit object/void pointer conversion for equality and assignment; one-evaluation compound and update nodes with distinct result and computation types; typed C11 conditional values and selection; and typed C11 `for`, `while`, `do`, and `switch` statements.

An `IF` node retains a required scalar condition after the ordinary default conversions, one required body, and an optional `else_body` that uses `CTOOL_C_AST_NONE` when absent. Both bodies precede the node in statement postorder. Recursive statement parsing binds `else` to the nearest unmatched `if`, and no control-flow target is embedded before IR lowering. A `FOR` node uses `CTOOL_C_AST_NONE` when its initializer is omitted and otherwise names a present expression or declaration statement. It also retains an optional scalar condition after the ordinary default conversions, an optional converted iteration expression, and one required body statement. `WHILE` and `DO` nodes retain one converted scalar condition and one body. A `DO` body and its expressions precede the condition and loop node, which preserves the source's post-test order. Later IR lowering must send a `continue` in that body to the condition, not back to the body start. A `SWITCH` node retains the integer-promoted condition and its required body. Each `CASE` retains one folded integer constant converted to that promoted type, while `DEFAULT` retains only its body. Nested switch contexts isolate their values and default labels. Duplicate values are rejected after conversion, as required by [C11 draft N1570, 6.8.4.2](https://www.open-std.org/jtc1/sc22/wg14/www/docs/n1570.pdf). Null expression statements retain `CTOOL_C_AST_NONE`, while `BREAK` and `CONTINUE` are targetless leaves whose nearest target remains a lowering decision. A switch supplies a `break` context but not an iteration context. One loop scope covers a declaration initializer, both expression clauses, and the body, then expires. A block binding keeps the object's identity and indexes the semantic initializer forest. Initialized automatic objects use converted `EXPRESSION` records and honor C's point of declaration. Supported block-scope static objects use `ZERO`, target-converted `INTEGER`, narrow character-array `STRING`, direct narrow-string `ADDRESS`, or recursive `LIST` records. List edges name direct array elements or structure members in postorder, omitted subobjects remain implicit zero, and direct unknown-bound arrays receive private completed types. Automatic aggregate lists, designators, and union or Cupid class lists remain deferred. Function definitions retain their own declared type, `inline`, storage, parameters, body, and dual location separately from the canonical entity summary.

Identifier labels use their C11 function scope and a namespace separate from ordinary identifiers. Each definition owns a contiguous slice of canonical label records. The first definition or `goto` creates the identity, and the label definition later fills its source locations and defining `LABEL` statement. A `LABEL` node owns its body as an ordinary postorder child. A `GOTO` node is a leaf with a semantic reference to the canonical label. That reference may point forward or backward and is not an AST child. Duplicate definitions fail immediately, and unresolved targets fail when the function ends. Computed goto and GNU label addresses remain explicit extension boundaries. The C11 rule against jumping into a variably modified object's scope needs no separate check yet because variable-length arrays are still unsupported.

Pointer comparisons apply the ordinary lvalue, array, and function conversions. Equality accepts pointers to compatible types, plus an object pointer paired with a qualified or unqualified `void` pointer. The object operand receives an explicit `CTOOL_C_CONVERSION_POINTER` node in the latter case. Relational comparison requires compatible object pointer types and produces target signed `int`; C object types may be complete, incomplete, or `void`, while function types remain outside that constraint. Cupid follows [WG14 N2209's proposed correction](https://www.open-std.org/jtc1/sc22/wg14/www/docs/n2209.htm) for atomic referents: comparison and subtraction accept atomic, qualified, or unqualified versions of compatible types, including mixed atomic pairs. Assignment separately requires both referents to be atomic or neither to be.

The same pointer conversion kind represents assignment conversion between object and `void` pointers when the destination referent retains every source qualifier and both referents are atomic or neither is. Atomic-qualified `void` is still outside ADR 0013's layout contract, so the both-atomic object/void assignment allowed by N2209 remains unsupported. Pointer/integer equality stays deferred when the integer operand might be a zero-valued integer constant expression. Direct nonzero constants, lvalue-converted objects, calls, assignments, and updates are known not to qualify and receive constraint diagnostics. A checked iterative AST walk finds those disqualifiers beneath supported arithmetic and casts without recursing through long expression chains. For logical short-circuit operations it checks the always-evaluated left operand and leaves the potentially skipped right operand unresolved, so a call in an unevaluated operand is not falsely rejected. A possible `(void *)` cast of an unresolved zero-valued integer constant expression is likewise deferred before an incorrect object/void conversion can be published. Literal `0` remains unsupported with the other candidate constant-expression forms rather than receiving a one-token shortcut.

A `CONDITIONAL` expression owns three ordered postorder children: the converted condition, the nonzero arm, and the zero arm. It never yields an lvalue. Integer arms use the Cupid i386 usual arithmetic conversions. Compatible pointer arms build a composite referent and merge its immediate qualifiers; object/`void` pairs produce an appropriately qualified `void` referent. N2209 atomic qualification participates in that merge, while atomic-qualified `void` remains the explicit ADR 0013 boundary. Array and function arms decay before result selection. Two `void` arms produce `void`. Later IR lowering must evaluate the condition, observe C11's sequence point, evaluate only the selected arm and its conversions, and join the result. Floating, compatible aggregate, and unresolved null-pointer forms remain valid but deferred. The integer constant-expression path uses the same right association and evaluates only the selected arm while still parsing both.

Floating controlling expressions for loops and `if` remain valid but deferred until floating conversions exist, whereas aggregate and `void` conditions violate the scalar constraint. A switch condition must have integer type, so pointer, floating, aggregate, and `void` controls are rejected. Comma expressions remain a deliberate boundary. Automatic aggregate lists, designated and union static lists, file-scope initialization, non-string address and floating constants, static-data allocation and relocations, computed goto and GNU label addresses, floating and null-pointer-constant conversions, translation-unit finalization, linear IR, code generation, and production compiler ownership remain later gates.
