# Use ELF32 relocatable objects as the toolchain interchange

CupidC and CupidASM will emit deterministic ELF32 `ET_REL` objects, and CupidLD will consume them while preserving the subset of GNU linker-script behavior used by Cupid OS. This supports relocations, symbols, mixed-toolchain migration, and existing image layout without committing Cupid OS to either flat-binary-only compilation or an unrelated custom object format.

The shared `toolchain/elf32` module now owns this interchange behind two operations: `ctool_elf32_write` accepts a complete semantic description, and `ctool_elf32_read` returns a bounded typed view. Callers describe ordered payload sections, symbols, and relocations; they do not choose ELF file offsets, string offsets, serialized indices, metadata tables, or padding. A stateful ELF builder was rejected because frontend parser state and partial object mutation would leak through the format seam, while exposing raw ELF tables was rejected because every caller would have to repeat layout and validation policy.

The v1 writer emits little-endian i386 `ET_REL` with caller sections in order, one generated `SHT_REL` section per relocated target, then `.symtab`, `.strtab`, `.shstrtab`, and the section-header table. It stable-partitions local symbols before global and weak symbols, preserves order within each partition, remaps relocation references, deduplicates strings by first occurrence, zero-fills padding, and converts explicit signed addends into the four-byte implicit fields required by `SHT_REL`. It supports `PROGBITS`/`NOBITS`, local/global/weak bindings, undefined/defined/absolute/common storage, and `R_386_32`/`R_386_PC32`. V1 payload flags are limited to write, allocate, execute, merge, strings, TLS, and exclude; flags requiring link, info, group, or compression metadata are rejected until the semantic interface can represent them. Merge/string entry invariants are validated. Equal semantic inputs produce byte-identical output without time, path, locale, allocator-address, or host dependence.

The writer deliberately accepts only the canonical supported subset and rejects ambiguous or invalid models. The reader is more permissive: it validates all bounds and the represented known cross-table semantics without packed-structure casts, but preserves well-formed unfamiliar sections and relocation types for an objdump-class consumer, including raw section addresses and flags. Unknown relocations have no decoded addend. Primary serialized reader references are explicitly named as file-index values; raw type-dependent `sh_link`/`sh_info` fields document the same domain, and reader section/symbol arrays include null/generated entries. Writer indices address only the caller's semantic arrays. Referenced string offsets are sorted and resolved with one forward pass over each string table, bounding hostile-name work by table bytes plus sorting the references. Extended section indices, multiple symbol tables, program headers, non-i386 domains, and executable layout remain outside this module. Existing CupidC/CupidASM fixed-address `ET_EXEC` output remains behavior-compatible until their frontends migrate, and final executable layout remains CupidLD's responsibility.

Writer output is transactional and reader metadata is job-arena owned. A failed write restores the empty output buffer; a failed read rewinds temporary arena state, zeros its result, and retains a stable structured diagnostic. Reader names and payload views borrow the source image. The module depends only on the platform-neutral toolchain job, arena, source, diagnostic, and buffer contracts, so the same implementation is built by the hosted and kernel adapters.
